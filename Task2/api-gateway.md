# API Gateway для DSP-интеграции: дизайн

Документ описывает API-слой для приёма bid requests от новой DSP-площадки: маршрутизацию, ограничение частоты запросов, аутентификацию, паттерн Circuit Breaker и мониторинг времени отклика.

---

## 1. Роль Gateway на bid-пути

API Gateway для DSP-трафика спроектирован как тонкий слой маршрутизации, а не как «умная труба» с тяжёлой агрегацией данных. Причина: весь путь запроса, включая прохождение через Gateway, укладывается в бюджет задержки не более 80 ms. Любая дополнительная логика на Gateway (агрегация из нескольких сервисов, сложные трансформации) съедает часть этого бюджета до того, как запрос дойдёт до Bidding & Delivery Service.

Функции, которые Gateway выполняет на bid-пути: аутентификация, rate limiting, маршрутизация, Circuit Breaker, логирование задержки. Функции, которые Gateway не выполняет: агрегация данных из нескольких сервисов, трансформация полезной нагрузки сверх необходимого протокольного преобразования, бизнес-логика аукциона.

## 2. Маршрутизация

| Входящий путь | Целевой сервис | Протокол |
|---|---|---|
| `POST /openrtb2/bid` | Bidding & Delivery Service | REST снаружи, gRPC внутри |
| `GET /openrtb2/status` | Bidding & Delivery Service (health check) | REST |
| `POST /api/v1/campaigns/**` | Advertiser Dashboard Service | REST |
| `GET /api/v1/reports/**` | Advertiser Dashboard Service | REST |

Gateway выполняет преобразование протокола: DSP-партнёр отправляет REST/JSON-запрос по OpenRTB, а внутри системы Gateway обращается к Bidding & Delivery Service по gRPC для минимизации задержки внутреннего вызова.

## 3. Аутентификация

DSP-партнёр аутентифицируется по API-ключу, выданному при подключении интеграции, а не по пользовательскому JWT: это server-to-server канал без пользовательской сессии, аналогично паттерну Public API для внешних партнёрских интеграций.

Gateway проверяет валидность API-ключа до маршрутизации запроса к Bidding & Delivery Service. Если ключ невалиден или отозван, Gateway немедленно возвращает ошибку авторизации, не тратя латентностный бюджет на обращение к внутреннему сервису.

Рекламодатель, обращающийся через веб-интерфейс, аутентифицируется отдельно по JWT-токену, полученному при входе в личный кабинет. Проверка JWT происходит централизованно на Gateway по тому же принципу: сервисы не дублируют проверку токена, а доверяют результату проверки Gateway.

## 4. Rate limiting

Ограничение частоты запросов реализовано на уровне Gateway, а не на уровне Bidding & Delivery Service, чтобы лимит применялся до того, как запрос израсходует ресурсы внутреннего сервиса и латентностный бюджет.

Параметры лимитов ниже представляют собой обоснованную инженерную оценку для RTB-интеграции, поскольку условия проекта не задают конкретных числовых значений лимитов:

| Канал | Лимит | Обоснование |
|---|---|---|
| DSP-партнёр (по API-ключу) | Согласованная с партнёром квота на основе целевого RPS с запасом (например, 15 000 запросов в секунду при целевых 18 000 на первом этапе с учётом доли трафика от одного партнёра) | Защита от превышения согласованной пропускной способности одним партнёром и от случайного шторма запросов |
| Рекламодатель (по JWT) | 100 запросов в минуту на пользователя | Защита личного кабинета от brute force и случайных циклов в клиентском коде, аналогично типовому лимиту для пользовательских операций |
| Неаутентифицированные запросы (по IP) | 50 запросов в минуту | Защита от DDoS до прохождения аутентификации |

Лимит на уровне Gateway работает глобально: партнёр, исчерпавший квоту на одном пути, не может продолжить нагружать систему через другой эндпоинт, поскольку Gateway видит весь входящий трафик, а не только трафик одного сервиса.

## 5. Circuit Breaker

Circuit Breaker для вызовов от Gateway к Bidding & Delivery Service защищает от каскадного отказа: если сервис перестаёт отвечать в срок или начинает возвращать ошибки, Gateway перестаёт направлять ему запросы и возвращает fallback вместо того, чтобы копить очередь зависших запросов.

**Состояния:**

- **Closed (закрыто).** Запросы проходят штатно, Gateway отслеживает долю ошибок и таймаутов.
- **Open (открыто).** При превышении порога ошибок Gateway прекращает отправлять запросы к Bidding & Delivery Service и немедленно возвращает fallback-ответ DSP-партнёру, не дожидаясь таймаута реального вызова.
- **Half-Open (полуоткрыто).** После паузы Gateway пропускает ограниченное число тестовых запросов. Если они успешны, цепь снова замыкается; если нет, возвращается в открытое состояние.

**Fallback при разомкнутой цепи.** DSP-партнёру возвращается пустой bid response (отказ от участия в аукционе), а не ошибка сервера: для RTB-протокола это штатный сценарий не участвовать в конкретном аукционе, и он не увеличивает счётчик ошибок на стороне партнёра так, как это сделал бы HTTP 500.

**Реализация на уровне Gateway, а не внутри Bidding & Delivery Service.** Circuit Breaker в этом канале защищает Gateway и партнёра от бесполезного ожидания уже деградировавшего сервиса, а не сам сервис от партнёра. Это соответствует принципу: Circuit Breaker реализуется на стороне потребителя вызова, а не провайдера, поскольку его задача состоит в том, чтобы не тратить ресурсы на заведомо неуспешные попытки и обеспечить контролируемое поведение вместо того, чтобы решение об отключении принимал сам перегруженный сервис.

## 6. Мониторинг времени отклика

Gateway логирует каждый запрос с полями, необходимыми для расследования нарушений SLA перед DSP-партнёром: временная метка, идентификатор запроса, latency Gateway, latency вышестоящего сервиса (Bidding & Delivery Service), статус ответа, идентификатор партнёра.

Пример записи лога:

```
{
  "timestamp": "2027-03-15T10:30:15.123Z",
  "request_id": "req_a1b2c3",
  "method": "POST",
  "path": "/openrtb2/bid",
  "status": 200,
  "latency_ms": 62,
  "upstream_service": "bidding-delivery-service",
  "upstream_latency_ms": 48,
  "dsp_partner_id": "dsp_main",
  "circuit_breaker_state": "closed"
}
```

Раздельное логирование `latency_ms` (полное время, видимое партнёром) и `upstream_latency_ms` (время внутри Bidding & Delivery Service) позволяет отличить деградацию самого Gateway от деградации сервиса ставок при разборе нарушения SLA.

Метрика P95 и P99 latency по этому логу должна отслеживаться непрерывно и сверяться с контрактным порогом не более 80 ms: приближение к порогу является сигналом для эскалации до фактического нарушения SLA и штрафа со стороны DSP-партнёра.

## 7. Реализация

Выбор конкретной технологии Gateway зависит от требований к производительности, поддержки gRPC для внутренней маршрутизации, декларативной конфигурации и готовых модулей для аутентификации, rate limiting, Circuit Breaker и логирования. Любая из распространённых реализаций (реверс-прокси с расширениями, специализированный прокси для микросервисной архитектуры, управляемое решение) способна закрыть эти требования технически; выбор конкретного продукта зависит от технологического стека и операционного опыта команды AdScale и не фиксируется этим документом.
