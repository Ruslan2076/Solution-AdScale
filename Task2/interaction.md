# Схема взаимодействия: протоколы и паттерны

Документ определяет протоколы для трёх типов взаимодействия в RTB-архитектуре AdScale: внутреннего синхронного, внешнего с DSP-партнёром и потоковой обработки событий. Каждый выбор обоснован по критериям выбора протокола.

---

## 1. Внутреннее синхронное взаимодействие: gRPC

### Критерии выбора

Пять вопросов для выбора между REST и gRPC:

| Критерий | Ответ для внутренних вызовов AdScale |
|---|---|
| Результат нужен немедленно? | Да, для вызовов внутри Advertiser Dashboard (запрос баланса, запрос отчёта) |
| Допустимая задержка | Единицы и десятки миллисекунд |
| Критичность потери сообщения | Не критична для одиночного синхронного вызова, но важна предсказуемость задержки |
| Направление потока данных | Point-to-point, один вызывающий, один вызываемый сервис |
| Масштаб | Внутренний трафик, не рассчитан на десятки тысяч запросов в секунду для этих конкретных вызовов |

### Обоснование gRPC для Bidding и Campaign

Для внутренней коммуникации между сервисами gRPC предпочтителен по умолчанию: бинарный формат Protocol Buffers компактнее и быстрее в сериализации, чем текстовый JSON, а транспорт HTTP/2 даёт мультиплексирование запросов в одном соединении без блокировки очереди. При частых вызовах с небольшим объёмом данных (проверка баланса, запрос статуса) HTTP-заголовки REST создают накладные расходы на каждый запрос, которых у gRPC нет.

Дополнительный довод для AdScale: сервисы написаны на разных языках (Go/C++ для Bidding, Python для Analytics, Node.js для Dashboard, Go для Financial). gRPC обеспечивает типобезопасное взаимодействие между разнородными технологическими стеками через общие proto-контракты, без ручного согласования формата JSON между командами.

Применяется для:

- Advertiser Dashboard Service → Financial Service (запрос текущего баланса)
- Advertiser Dashboard Service → Analytics Service (запрос отчёта)

### Почему не REST для этих вызовов

REST остаётся оправданным выбором для внешних клиентов и простых CRUD-операций, но при жёстких требованиях к задержке между сервисами и частых вызовах с небольшим объёмом данных гRPC обычно эффективнее за счёт бинарного протокола и меньшего overhead на каждый запрос. Внутренние вызовы Dashboard к Financial и Analytics относятся именно к этой категории: частые, короткие, между доверенными внутренними сервисами, без нужды в браузерной совместимости.

### Где всё же используется REST

REST остаётся протоколом для внешних клиентов: рекламодатель обращается к Advertiser Dashboard через API Gateway по REST, поскольку это браузерный клиент и REST не требует от фронтенда генерации кода из proto-схемы. Аналогично REST используется для доступа рекламодателя из веб-интерфейса.

## 2. Внешнее взаимодействие с DSP-партнёром: REST/OpenRTB

Условия проекта прямо задают протокол интеграции с DSP-партнёром: OpenRTB поверх HTTPS. Это отраслевой стандарт RTB-индустрии, не описанный в материалах курса, но соответствующий общей рекомендации использовать REST для внешнего API партнёров: партнёры интегрируются с разными технологическими стеками, и открытый HTTP-протокол с документированным контрактом остаётся самым универсальным вариантом для внешней интеграции.

Особенность этого канала: жёсткий бюджет задержки не более 80 ms целиком, включая сетевой путь до AdScale, обработку и ответ. Это не типовой REST-сценарий с толерантностью к задержке в сотни миллисекунд, а причина, по которой API Gateway на этом пути должен оставаться максимально тонким слоем маршрутизации без тяжёлой агрегации данных.

## 3. Потоковая обработка событий: Kafka

### Критерии выбора между очередью сообщений и потоком событий

События кликов, показов и результатов аукциона (`BidWon`, `ImpressionRecorded`, `ClickRecorded`) представляют собой факты о том, что уже произошло, а не команды, которые нужно выполнить. У каждого события несколько независимых потребителей: Event Ingestion (запись событий), Financial (списание средств), Analytics (агрегация). Это классический сценарий Publish/Subscribe с моделью данных на основе событий, а не команд.

### Обоснование Kafka

По критериям выбора брокера для этого потока:

- **Направление потока данных.** Один продюсер (Bidding & Delivery), несколько независимых консьюмеров. Каждый из них должен получить собственную копию события и обрабатывать её в своём темпе, без конкуренции за сообщение.
- **Критичность потери сообщения.** Финансовые последствия событий `BidWon` делают потерю сообщения недопустимой: списание средств не должно быть пропущено. Гарантия доставки Kafka (минимум один раз по умолчанию, ровно один раз при использовании транзакций) закрывает это требование.
- **Масштаб.** Целевая нагрузка 50 000 RPS означает потенциально десятки и сотни тысяч событий кликов и показов в секунду. Это диапазон, где Kafka является единственным из рассмотренных брокеров с оценкой «отлично» по пропускной способности свыше 100 000 сообщений в секунду.
- **Повторное чтение и аудит.** Kafka хранит события согласно политике хранения и позволяет консьюмеру перечитать поток с любого смещения. Это востребовано для Analytics (переобработка исторических данных) и для журнала аудита финансовых операций.

По прямому сравнению сценариев: event streaming с несколькими консьюмерами получает у Kafka оценку «отлично», у RabbitMQ и Artemis эта оценка «плохо», поскольку у них нет повторного чтения и естественной поддержки множественных независимых подписчиков на один и тот же поток.

### Почему не RabbitMQ или Artemis

RabbitMQ и Artemis сильны там, где нужна классическая очередь задач с конкурирующими потребителями (Competing Consumers) и низкая операционная сложность, например для рассылки уведомлений. Для потока кликов и показов эта модель не подходит: события должны быть доставлены всем независимым консьюмерам, а не разобраны между воркерами как задачи, и должны сохраняться для повторного чтения, чего RabbitMQ и Artemis не предоставляют.

## 4. Нужен ли API Gateway

Да. Обоснование по критериям, применимым к AdScale:

- **Единая точка входа для трёх разных клиентов** (DSP-партнёр, конечный пользователь, рекламодатель) вместо того, чтобы каждый клиент обращался к своему сервису напрямую.
- **Централизованная аутентификация.** Без Gateway каждый сервис дублирует проверку токена или API-ключа DSP-партнёра, что для внешней интеграции с контрактными обязательствами создаёт риск рассинхронизации политик безопасности.
- **Единый rate limiting.** Ограничение частоты запросов DSP-партнёра должно применяться до маршрутизации к сервису, иначе партнёр, исчерпавший лимит на одном пути, сможет продолжить нагружать систему через другой.
- **Единая точка мониторинга задержки.** Обязательство по SLA перед DSP-партнёром требует видимости latency на входе в систему, а не только внутри Bidding Service.
- **Circuit Breaker для нисходящих сервисов.** Единая точка, где можно централизованно отслеживать деградацию downstream-сервисов и отдавать fallback, не встраивая эту логику в каждый сервис отдельно.

Детальный дизайн API Gateway для DSP-интеграции вынесен в отдельный документ `api-gateway.md`.
