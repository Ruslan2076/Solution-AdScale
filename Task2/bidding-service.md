# Bidding & Delivery Service: спецификация

Документ детализирует сервис ставок, выделенный из монолита AdScale первым по ADR-002. Описывает границы сервиса, API, зависимости от других сервисов и модель данных.

---

## 1. Границы сервиса

Bidding & Delivery Service отвечает за полный синхронный конвейер обработки одного рекламного показа: приём bid request, отбор кандидатов, проведение аукциона, формирование bid response и разметки баннера.

Три исходных компонента монолита (рекламный сервер, движок аукциона, сервис выдачи) объединены в один сервис, а не разделены на три. Причина: они образуют последовательный синхронный конвейер внутри одного запроса с общим бюджетом задержки не более 80 ms. Разнесение конвейера по сетевым границам внутри единого латентностного бюджета складывает задержки сетевых вызовов друг с другом и создаёт риск распределённого монолита.

**Сервис отвечает за:**

- Приём и валидацию bid request от DSP-партнёра
- Отбор кандидатов на показ по данным таргетинга
- Проведение аукциона: взвешивание ставок, применение бизнес-правил, определение победителя
- Формирование bid response и HTML/JS-разметки баннера
- Проверку остатка рекламного бюджета перед показом ставки
- Публикацию фактов о результате показа (выигрыш аукциона, показ, клик) для остальной системы

**Сервис не отвечает за:**

- Управление кампаниями, бюджетами и ставками (это Advertiser Dashboard Service)
- Списание средств и ведение балансов (это Financial Service)
- Долговременное хранение событий кликов и показов (это Event Ingestion Service)
- Построение отчётов и агрегатов (это Analytics Service)

## 2. API

### 2.1 Внешний API (DSP-партнёр, через API Gateway)

Внешняя интеграция построена на протоколе OpenRTB, указанном как обязательный в условиях проекта. OpenRTB является отраслевым стандартом протокола для RTB-аукционов; сам стандарт не описан в материалах курса, это отраслевая практика.

```
POST /openrtb2/bid
Content-Type: application/json
```

Запрос содержит идентификатор аукциона, параметры показа (размер баннера, площадка, геолокация пользователя), таймаут ответа. Ответ содержит цену ставки, идентификатор объявления-победителя, разметку креатива или пусто, если сервис отказывается от участия в аукционе.

### 2.2 Внутренний API (gRPC)

Внутренние вызовы к Bidding Service отсутствуют в hot path по конструкции: остальные сервисы не обращаются к Bidding Service синхронно, чтобы не создавать зависимость, способную замедлить критичный путь. Единственный внутренний вызов, не входящий в hot path:

```
service BiddingAdmin {
  rpc GetCampaignSyncStatus(SyncStatusRequest) returns (SyncStatusResponse);
  rpc InvalidateCache(InvalidateCacheRequest) returns (InvalidateCacheResponse);
}
```

Используется Advertiser Dashboard Service или службой эксплуатации для диагностики задержки синхронизации данных о кампаниях и принудительного сброса кэша.

## 3. Зависимости от других сервисов

| Зависимость | Направление | Тип | Обоснование |
|---|---|---|---|
| Advertiser Dashboard Service | Входящая, асинхронная | Kafka, события `CampaignUpdated`, `BudgetUpdated`, `BidStrategyUpdated` | Bidding Service не читает Campaign DB напрямую по принципу database-per-service. Собственная read-модель строится асинхронно из потока событий |
| Financial Service | Входящая, асинхронная | Kafka, события `BalanceChanged`, `BudgetExhausted` | Проверка остатка бюджета идёт по локальному кэшированному счётчику, а не синхронным вызовом, чтобы не вносить сетевую зависимость в путь с бюджетом 80 ms |
| Kafka (брокер) | Исходящая, асинхронная, publish | Топики `BidWon`, `ImpressionRecorded`, `ClickRecorded` | Результаты показа передаются остальной системе без ожидания подтверждения обработки |
| Bidding Cache (Redis) | Синхронная, cache-aside | Внутренняя инфраструктура сервиса | Быстрое чтение данных таргетинга и кампаний без обращения к БД на каждый bid request |
| Bidding DB (PostgreSQL) | Синхронная, чтение | Внутренняя инфраструктура сервиса | Источник для промаха кэша; read-модель, а не источник истины по кампаниям |

Важное архитектурное решение: у Bidding Service нет ни одной синхронной зависимости от других доменных сервисов на hot path. Все зависимости от Advertiser Dashboard и Financial реализованы асинхронно через Kafka. Это прямое следствие принципа из паттернов Retry и Circuit Breaker: зависимость, способная замедлиться или отказать, не должна находиться в пути с жёстким бюджетом задержки без механизма быстрой деградации.

## 4. Модель данных

Bidding DB хранит read-оптимизированную копию данных, необходимых для быстрого принятия решения об участии в аукционе. Схема ниже отражает типовую структуру для этой предметной области; конкретные поля не заданы условиями проекта и определены по общим принципам предметной области.

**campaigns** (реплика из Advertiser Dashboard через события)
- `campaign_id` (PK)
- `advertiser_id`
- `status` (active, paused, budget_exhausted)
- `targeting_rules` (JSON: гео, устройство, площадка)
- `updated_at`

**bids**
- `bid_id` (PK)
- `campaign_id` (FK)
- `bid_price`
- `bid_strategy` (fixed, dynamic)
- `updated_at`

**budget_cache** (локальный кэш остатка бюджета, источником истины является Financial Service)
- `campaign_id` (PK)
- `remaining_budget`
- `last_synced_at`
- `sync_lag_ms` (для мониторинга отставания от Financial Service)

**auction_log** (краткосрочный журнал для отладки и аудита, не для аналитики)
- `auction_id` (PK)
- `winning_bid_id`
- `winning_price`
- `latency_ms`
- `created_at`, TTL ограничен несколькими днями

Аналитические и долгосрочные данные о показах и кликах в этой базе не хранятся: это зона ответственности Event Ingestion Service, наполняемого через Kafka, а не Bidding DB.

## 5. Резервные стратегии при недоступности зависимостей

Поскольку у сервиса нет синхронных внешних зависимостей на hot path, отказ Advertiser Dashboard или Financial Service не блокирует аукцион напрямую, а деградирует его так:

- **Устаревшие данные о кампаниях.** Если поток событий `CampaignUpdated` отстаёт или недоступен, Bidding Service продолжает работать по последним известным данным из кэша и Bidding DB. Риск: участие в аукционе по устаревшей ставке или таргетингу до восстановления синхронизации.
- **Устаревший остаток бюджета.** Если поток `BalanceChanged` отстаёт, проверка идёт по последнему известному значению `remaining_budget`. Риск: небольшой перерасход бюджета в пределах окна отставания синхронизации. Компенсируется тем, что Financial Service обрабатывает списания идемпотентно и может дополнительно блокировать кампанию при обнаружении перерасхода постфактум.
- **Полный отказ Bidding Cache (Redis).** Резервным путём становится прямое чтение из Bidding DB (read-replica), с более высокой, но всё ещё приемлемой задержкой.
- **Дефолтная ставка.** Если для кандидата нет валидных данных ни в кэше, ни в БД, сервис не выставляет ставку по этому кандидату, а не подставляет произвольное дефолтное значение: ставка вслепую создаёт финансовый риск, тогда как отсутствие ставки означает лишь упущенный показ.
