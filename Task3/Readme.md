# Задание 3: Данные, масштабирование и отказоустойчивость

Стратегия работы с данными в микросервисной архитектуре AdScale: выбор базы данных для каждого сервиса, масштабирование хранилищ, кэширование, потоковая обработка событий и отказоустойчивость данных. Документы опираются на границы сервисов, зафиксированные в `TO-BE.md` (задание 1), и на дизайн Bidding & Delivery Service из задания 2.

## Структура документов

- **database-strategy.md**: выбор типа базы данных для каждого из пяти сервисов (Bidding, Campaign, Stats, Analytics, Financial) по пяти критериям: тип данных, консистентность, профиль нагрузки, масштабирование, операционная зрелость команды.
- **scaling-strategy.md**: какие сервисы нуждаются в Master-Slave репликации, какие данные и по какому ключу шардируются, обоснование read-replica для Analytics, применение CQRS на границах сервисов.
- **caching-strategy.md**: что кэшируется в Redis перед Bidding & Delivery Service, размещение кластера, TTL и инвалидация по типам данных, прогрев кэша.
- **streaming-architecture.md**: топики Kafka, схема событий, группы потребителей, политика хранения.
- **data-resilience.md**: RPO и RTO для каждого сервиса, стратегия резервного копирования, соответствие целевой архитектуры принципам 12-факторных приложений.

## Ключевой принцип, объединяющий документы

Большинство сервисов AdScale хранят не источник истины, а read-модель, восстановимую из потока событий Kafka. Источников истины в системе всего два: Campaign DB (кампании, бюджеты, ставки) и Financial DB (балансы, транзакции). Bidding DB, Stats DB и Analytics DB построены как проекции этих источников через асинхронные события, поэтому для них резервное копирование, консистентность и требования к RPO принципиально мягче, чем для двух источников истины. Это разделение проходит через все пять документов: от выбора типа базы данных до стратегии восстановления после сбоя.
