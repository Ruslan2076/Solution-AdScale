# Streaming Architecture: потоковая обработка событий

Документ описывает архитектуру потоковой обработки событий AdScale: топики Kafka, группы потребителей и политику хранения.

---

## 1. Топики Kafka

Топики сгруппированы по источнику события (по продюсеру), а не по каждому отдельному типу события, по аналогии с типовой практикой, где один топик объединяет несколько связанных типов событий одного домена (`TransferCreated`, `TransferCompleted`, `TransferFailed`), а не создаёт отдельный топик под каждый тип.

| Топик | Продюсер | Типы событий |
|---|---|---|
| `adscale.bidding.events` | Bidding & Delivery Service | `BidWon`, `ImpressionRecorded`, `ClickRecorded` |
| `adscale.campaign.events` | Advertiser Dashboard Service | `CampaignUpdated`, `BudgetUpdated`, `BidStrategyUpdated` |
| `adscale.financial.events` | Financial Service | `BalanceChanged`, `BudgetExhausted` |

Разделение по источнику, а не по потребителю, соответствует принципу событий как фактов: продюсер публикует то, что произошло в его домене, не зная заранее, кто и сколько потребителей будет это читать.

## 2. Схема событий: Avro или JSON

Рекомендация: Avro со Schema Registry.

- **Объём трафика.** При целевой нагрузке 50 000 RPS поток `adscale.bidding.events` потенциально достигает десятков тысяч сообщений в секунду. Компактный бинарный формат Avro снижает нагрузку на сеть и диск кластера Kafka по сравнению с текстовым JSON, аналогично тому, как для внутренних синхронных вызовов был выбран компактный gRPC/Protobuf вместо REST/JSON именно по причине высокой частоты вызовов (обоснование в `interaction.md`, задание 2).
- **Множественные независимые потребители одного контракта.** У `adscale.bidding.events` минимум три потребителя (Event Ingestion, Financial, Analytics), каждый из которых обязан одинаково понимать формат события, особенно поле цены ставки в `BidWon`, от которого зависит списание средств. Schema Registry принудительно проверяет совместимость схемы при публикации, не позволяя продюсеру выпустить breaking change, который потребители не готовы обработать.
- **Компенсация отсутствия выделенного бизнес-аналитика и SRE.** В команде AdScale нет ни выделенного бизнес-аналитика, ни SRE-инженера. Меньше людей физически способны вручную отследить рассинхронизацию контракта между сервисами при ручной проверке JSON, поэтому автоматизированная проверка схемы даёт больше пользы, а не меньше.

Оговорка: Schema Registry является дополнительным компонентом инфраструктуры, который нужно разворачивать и поддерживать. Это реальная операционная нагрузка на команду без SRE, аналогичная риску, отмеченному для ClickHouse в `database-strategy.md`. Если операционные ресурсы окажутся более узким местом, чем пропускная способность сети, можно временно остаться на JSON с ручным версионированием полей и вернуться к Avro при выделении SRE.

## 3. Группы потребителей

Каждый downstream-сервис читает нужный ему топик в собственной группе потребителей, независимо от остальных. Это прямое применение паттерна Publish/Subscribe, где у каждого события несколько независимых потребителей, каждый в своей группе, читающих поток параллельно и без конкуренции за сообщения.

| Топик | Consumer Group | Потребитель | Назначение |
|---|---|---|---|
| `adscale.bidding.events` | `stats-ingestion` | Event Ingestion Service | Запись сырых событий в Stats DB |
| `adscale.bidding.events` | `financial-charging` | Financial Service | Списание средств по `BidWon`, идемпотентно |
| `adscale.bidding.events` | `analytics-aggregation` | Analytics Service | Построение агрегатов для отчётности |
| `adscale.campaign.events` | `bidding-campaign-sync` | Bidding & Delivery Service | Обновление локальной read-модели кампаний |
| `adscale.financial.events` | `bidding-budget-sync` | Bidding & Delivery Service | Обновление кэшированного остатка бюджета |

Механика распределения партиций внутри одной consumer group (какой именно инстанс сервиса читает какую партицию) является деталью реализации Kafka и в проектной документации фиксируется только состав групп и их назначение.

**Идемпотентность на стороне потребителя.** Consumer group `financial-charging` обязана обрабатывать повторную доставку одного и того же события без повторного списания, поскольку Kafka по умолчанию гарантирует доставку не менее одного раза. Механизм идемпотентности (`idempotency_key = bid_id + operation_type`) уже описан в `interaction.md` (задание 2) и здесь не дублируется.

**Dead Letter Queue.** Для `financial-charging` предусмотрен DLQ: если событие `BidWon` не удаётся обработать после ограниченного числа попыток, например из-за поломанной структуры сообщения, оно уходит в отдельный топик `adscale.financial.events.dlq` вместо того, чтобы блокировать обработку остального потока или теряться бесследно.

## 4. Политика хранения

Kafka хранит события согласно настраиваемой политике хранения, от часов до неограниченного срока, в отличие от классической очереди сообщений, где сообщение удаляется сразу после обработки. Выбор политики для каждого топика ниже основан на роли данных в системе.

| Топик | Рекомендуемая политика хранения | Обоснование |
|---|---|---|
| `adscale.bidding.events` | 30 дней | Kafka хранит domain events после доставки специально для возможности провести финансовый аудит и восстановить полную историю операции. Для AdScale это события, напрямую связанные со списанием средств, поэтому требуют возможности перечитать историю при расследовании расхождений в балансах |
| `adscale.campaign.events` | 7 дней | Эти события в первую очередь нужны Bidding & Delivery Service для построения текущего состояния read-модели, а не для длительного аудита. Более короткое хранение достаточно для восстановления состояния при переразвёртывании сервиса |
| `adscale.financial.events` | 30 дней | Та же логика аудита, что и для `adscale.bidding.events`: изменения баланса должны быть прослеживаемы на сопоставимом горизонте |

**Возможное развитие: log compaction.** Для `adscale.campaign.events` и `adscale.financial.events` рассматривается альтернатива временной политике хранения: режим compacted-топика, при котором Kafka хранит не всю историю по времени, а только последнее значение по каждому ключу (например, по `campaign_id`) неограниченно долго. Это избавило бы от необходимости выбирать конкретный срок хранения для событий, единственная ценность которых, помимо аудита, в восстановлении актуального состояния read-модели. Решение сознательно не закладывается как основное в этом документе, поскольку требует отдельной проверки на соответствие требованиям аудита финансовых изменений, а не только удобства восстановления состояния.
