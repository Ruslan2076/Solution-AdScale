# Caching Strategy: стратегия кэширования

Документ определяет, что кэшируется в AdScale, где размещён Redis, какие TTL и стратегии инвалидации применяются к разным типам данных, и как организован прогрев кэша.

Кэширование применяется по паттерну Cache Aside: приложение сначала проверяет кэш, при промахе читает из базы данных и на обратном пути кладёт данные в кэш.

---

## 1. Что кэшируется

Кэшируются три типа данных, которые Bidding & Delivery Service читает на каждый bid request: кампании, бюджеты и ставки. У них принципиально разная скорость изменения, поэтому единой TTL-стратегии для всех трёх недостаточно.

| Данные | Как часто меняются | Кто инициирует изменение |
|---|---|---|
| Кампании (статус, правила таргетинга) | Редко: рекламодатель включает, приостанавливает кампанию или меняет таргетинг | Advertiser Dashboard Service, событие `CampaignUpdated` |
| Бюджеты (остаток бюджета кампании) | Очень часто: на каждый выигранный аукцион | Financial Service, событие `BalanceChanged` |
| Ставки (цена, стратегия) | Умеренно: рекламодатель меняет стратегию ставок | Advertiser Dashboard Service, событие `BidStrategyUpdated` |

Кэширование этих данных оправдано по обоим критериям целесообразности из источника: нагрузка на чтение здесь экстремально высокая (каждый из 18 000, затем 50 000 запросов в секунду читает эти данные), а сами данные по кампаниям и ставкам меняются далеко не при каждом чтении.

## 2. Где размещён Redis

Redis Cluster размещён как отдельная инфраструктурная единица перед Bidding DB, обслуживающая все горизонтально масштабируемые инстансы Bidding & Delivery Service, а не как локальный кэш внутри процесса каждого инстанса.

Причина: Bidding & Delivery Service масштабируется горизонтально независимо от остальной системы. Если бы кэш был локальным для каждого инстанса, во-первых, каждый новый инстанс при масштабировании стартовал бы с пустым кэшем именно в момент пиковой нагрузки, когда автомасштабирование и добавляет новые инстансы, во-вторых, инстансы могли бы разойтись по данным, если бы получали события синхронизации не строго одновременно. Общий Redis Cluster устраняет оба риска: кэш прогревается один раз для всех инстансов и остаётся согласованным между ними.

По механизму Redis Cluster минимальная конфигурация состоит из трёх сегментов, в каждом из которых есть главный узел, отвечающий за запись, и узел-реплика с копией данных. Это даёт кэшу собственную отказоустойчивость, не зависящую от отказоустойчивости Bidding DB.

## 3. TTL и стратегия инвалидации по типам данных

Общий принцип: чем критичнее актуальность данных, тем короче TTL и тем более точная нужна инвалидация. Ниже указаны ориентировочные значения TTL: конкретные числа не заданы условиями проекта и подобраны по общей логике критичности каждого типа данных, а не взяты из источника.

### Кампании: инвалидация по событию плюс TTL как страховка

Основная инвалидация происходит по событию: как только Advertiser Dashboard Service публикует `CampaignUpdated`, Campaign Sync Consumer внутри Bidding & Delivery Service обновляет запись в Redis немедленно, не дожидаясь истечения TTL. Это тот случай, где точность важна: если рекламодатель приостановил кампанию, дальнейшие ставки по ней недопустимы даже в течение короткого времени.

TTL выступает не основным механизмом, а страховкой на случай отставания или сбоя потока событий: средний TTL порядка 15 до 30 минут. Если событие синхронизации по какой-то причине не дошло, запись в кэше всё равно не проживёт дольше этого срока и будет перечитана из Bidding DB, что ограничивает окно потенциальной рассинхронизации.

### Бюджеты: обновление по событию, а не классическая инвалидация

Остаток бюджета кэшируется по несколько иной логике, чем кампании: это не пара «инвалидировать и перечитать при следующем запросе», а непрерывно обновляемая проекция. Каждое событие `BalanceChanged` от Financial Service сразу уменьшает закэшированное значение `remaining_budget`, вместо того чтобы удалять запись и заставлять следующий bid request идти в базу данных за актуальным числом: budget-проверка находится на самом горячем пути системы, и лишний поход в БД на каждое списание недопустим.

TTL для бюджета короткий, порядка одной до пяти минут, и играет ту же роль страховки: если поток событий `BalanceChanged` отстаёт дольше этого порога, это сигнал деградации синхронизации, а не просто устаревания кэша, и должен фиксироваться отдельно через метрику отставания (`sync_lag_ms`), описанную в `bidding-service.md` (задание 2).

### Ставки: инвалидация по событию, средний TTL

Аналогично кампаниям: изменение ставки или стратегии рекламодателем публикуется как `BidStrategyUpdated` и сразу обновляет запись в Redis. TTL здесь может быть чуть длиннее, чем у кампаний, порядка 30 до 60 минут, поскольку ставки меняются реже, чем включаются или приостанавливаются кампании.

### Почему не инвалидация по версии

Инвалидация по версии добавляет ключу метку версии данных и подходит там, где важно избежать одновременной записи разными узлами. В текущей архитектуре запись в Redis для всех трёх типов данных выполняет один логический источник, Campaign Sync Consumer внутри Bidding & Delivery Service, читающий единственный упорядоченный поток Kafka. Гонки записи, которую решает версионирование, здесь структурно не возникает, поэтому эта стратегия избыточна для данного сценария.

## 4. Прогрев кэша

Прогрев нужен по двум сценариям, характерным именно для этой архитектуры: холодный старт при деплое или рестарте и добавление новых инстансов Bidding & Delivery Service при автомасштабировании под растущую нагрузку.

**Что прогревается.** Список hot keys составляют данные по всем активным кампаниям (статус active), а не по всем кампаниям в системе: приостановленные и завершённые кампании не участвуют в аукционе и не должны занимать место в разогретом кэше.

**Порядок прогрева.**

1. Получить список идентификаторов активных кампаний из Bidding DB.
2. Для каждой кампании прочитать данные о таргетинге, ставке и остатке бюджета.
3. Записать данные в Redis с соответствующим TTL для каждого типа.
4. Повторить перед известными пиками нагрузки и при каждом добавлении нового инстанса или сегмента кластера.

**Почему прогрев выполняется батчами.** Поскольку прогрев читает те же данные, которые Bidding DB отдаёт под обычной боевой нагрузкой, job прогрева не должен сам создавать всплеск нагрузки на базу данных именно в момент, когда это особенно нежелательно, то есть непосредственно перед или во время пика. Прогрев выполняется небольшими порциями с ограничением скорости запросов к Bidding DB, а не одним массовым чтением всех активных кампаний сразу.

**Масштаб.** При росте числа одновременно активных кампаний до объёма, когда последовательный прогрев начинает занимать заметное время, прогрев переводится на очередь с несколькими параллельными обработчиками: один процесс формирует список ключей для прогрева, а несколько воркеров заполняют Redis параллельно, не превышая контролируемую скорость обращения к базе данных.
