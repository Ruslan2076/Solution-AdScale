# Database Strategy: выбор базы данных для каждого сервиса

Документ определяет стратегию данных для пяти сервисов AdScale по пяти критериям: тип базы данных, требования к консистентности, профиль нагрузки (access pattern), способ масштабирования и операционная зрелость команды.

---

## Сводная таблица

| Сервис | Тип базы данных | Обоснование |
|---|---|---|
| Сервис ставок (Bidding & Delivery) | Реляционная (PostgreSQL) с read-replica, за Redis-кэшем | Read-heavy профиль нагрузки: кампании и таргетинг читаются на каждый bid request, а пишутся редко, только при синхронизации из потока событий. Структура данных стабильная. Строгая консистентность не требуется: сервис работает по read-модели, допускающей eventual consistency |
| Сервис кампаний (Advertiser Dashboard) | Реляционная (PostgreSQL) | Классический CRUD с чёткой стабильной структурой (кампании, бюджеты, ставки). Нужна causal-консистентность: рекламодатель должен сразу видеть собственные изменения после сохранения. Умеренный объём чтений и записей, шардирование не требуется |
| Сервис статистики (Event Ingestion) | Колоночная (ClickHouse) | Write-heavy профиль на грани десятков тысяч событий кликов и показов в секунду при целевой нагрузке 50 000 RPS. Данные пишутся построчно, но читаются почти всегда агрегированно (счётчики за период), а не точечными запросами. Строчная реляционная БД при таком объёме повторяет проблему AS-IS: статистика создавала write-bottleneck на общей БД |
| Сервис аналитики (Analytics) | Колоночная (ClickHouse) | Тяжёлые аналитические запросы с агрегацией по миллионам событий для дашбордов рекламодателей. Реляционная БД хранит данные построчно и вынуждена перебирать всю таблицу для агрегатов; колоночная БД возвращает такой результат за секунды. Допустима eventual consistency: отчёт может немного отставать от реального времени |
| Сервис финансов (Financial) | Реляционная (PostgreSQL) | Строгая ACID-консистентность обязательна: списание средств не терпит потерь и рассинхронизации балансов. Классический профиль «много записей, ведущих к строкам в журнале транзакций, редкие чтения истории» хорошо ложится на PostgreSQL с master-slave репликацией |

## Детальное обоснование по критериям

### Сервис ставок

**Тип БД.** Реляционная база данных, а не документная или колоночная, поскольку структура данных о кампаниях и ставках чёткая и стабильная, а не меняется от записи к записи.

**Консистентность.** Bidding DB является read-моделью, построенной асинхронно из потока событий `CampaignUpdated`, `BudgetUpdated`. Источник истины по кампаниям находится в сервисе кампаний, поэтому строгая консистентность здесь не нужна и была бы избыточным требованием для копии данных.

**Профиль нагрузки.** Много чтений на каждый bid request, мало записей (только при получении события синхронизации). Это ровно случай, для которого оптимальное решение: read replica плюс кэш. Redis отдаёт закэшированные данные таргетинга без обращения к базе на большинстве запросов, PostgreSQL read-replica обслуживает промахи кэша.

**Масштабирование.** Горизонтальное масштабирование через добавление read-replica при росте нагрузки чтения. Шардирование не требуется, поскольку объём данных о кампаниях (не о событиях) остаётся умеренным даже при росте RPS.

**Операционная зрелость.** PostgreSQL хорошо знаком большинству backend-инженеров, что снижает риск при развёртывании первым сервисом по Strangler Fig, учитывая отсутствие выделенного SRE в штате.

### Сервис кампаний

**Тип БД.** Реляционная база данных: кампании, бюджеты и ставки представляют собой чётко структурированные сущности с постоянным набором полей, изменения которых происходят через явные CRUD-операции рекламодателя.

**Консистентность.** Causal-консистентность: рекламодатель должен немедленно видеть собственное изменение бюджета или ставки после сохранения, но глобальная синхронизация с другими читателями (например, с Bidding Service) происходит асинхронно через поток событий и не требует мгновенной согласованности.

**Профиль нагрузки.** Умеренный объём чтений и записей, характерный для административной панели, без пиковых нагрузок уровня hot path аукциона.

**Масштабирование.** На горизонте года вертикального масштабирования и стандартной репликации для отказоустойчивости достаточно. Специализированное решение не требуется.

**Операционная зрелость.** PostgreSQL не создаёт дополнительной операционной нагрузки для команды.

### Сервис статистики

**Тип БД.** Колоночная база данных. Данные о кликах и показах поступают потоком с высокой интенсивностью записи и почти не запрашиваются точечно: типовым запросом является счётчик показов или кликов по кампании за период, а не поиск одного конкретного события по идентификатору.

**Консистентность.** Eventual consistency: сервис читает события из Kafka асинхронно, задержка в доставке события в базу на доли секунды не критична для последующей отчётности.

**Профиль нагрузки.** Экстремально write-heavy: при целевой нагрузке 50 000 RPS количество событий кликов и показов может исчисляться десятками тысяч записей в секунду. Строчная реляционная база данных при такой нагрузке на запись повторяет ровно ту проблему, которая была выявлена в AS-IS-анализе: статистика создавала конкуренцию за ресурсы записи с транзакционной нагрузкой.

**Масштабирование.** Требует тщательной настройки партиционирования (естественным кандидатом для партиции является дата события) и шардирования при дальнейшем росте объёма.

**Операционная зрелость.** Это точка реального риска: колоночные БД сложнее в эксплуатации, чем PostgreSQL, а в штате AdScale нет выделенного SRE-инженера. Рекомендация: либо использовать управляемый (managed) вариант колоночной БД, снимающий часть операционной нагрузки, либо заложить обучение команды или найм профильного специалиста в план внедрения этого сервиса. Замалчивать этот риск неверно: выбор технически обоснован, но операционно он не бесплатный.

### Сервис аналитики

**Тип БД.** Колоночная база данных, отдельная от Stats DB, несмотря на одинаковую категорию технологии. Это два разных ограниченных контекста с раздельным владением данными: Stats хранит сырые события, Analytics строит предрасчитанные агрегаты и витрины для отчётности рекламодателей. Общая база данных на два сервиса нарушила бы принцип «база данных на сервис».

**Консистентность.** Eventual consistency: отчёт, отстающий от реального времени на минуты, не создаёт проблем для рекламодателя, просматривающего аналитику кампании.

**Профиль нагрузки.** Тяжёлые аналитические запросы с агрегацией по большим объёмам исторических данных, обслуживающие дашборд рекламодателя. Классический сценарий, для которого построчная реляционная БД слишком медленная.

**Масштабирование.** Аналогично Stats DB: партиционирование по времени, шардирование при росте объёма исторических данных.

**Операционная зрелость.** Тот же риск, что и у Stats DB, с той же рекомендацией по смягчению.

### Сервис финансов

**Тип БД.** Реляционная база данных. Финансовые операции требуют целостности связей между кампанией, транзакцией списания и балансом, что естественно выражается через таблицы со связями и ACID-транзакции.

**Консистентность.** Строгая консистентность обязательна без исключений: списание средств не должно теряться или задваиваться. Это прямое следствие идемпотентной обработки события `BidWon`, описанной в `interaction.md` (задание 2): идемпотентность защищает от повторной обработки, но сама операция списания остаётся строго согласованной ACID-транзакцией.

**Профиль нагрузки.** Много записей (списания, пропорциональные объёму выигранных аукционов), сравнительно редкие чтения (запрос текущего баланса из личного кабинета). Это соответствует случаю «много записей, мало чтений»: PostgreSQL с репликацией master-slave, где мастер оптимизирован под запись, а реплика обслуживает редкие запросы истории и баланса.

**Масштабирование.** Шардирование финансовых данных сознательно не рекомендуется на данном горизонте: строгая консистентность через несколько шардов резко усложняет транзакционную логику, а объём финансовых операций растёт пропорционально аукционам, а не независимо от них, и не достигает объёма, требующего шардирования, в отличие от Stats DB.

**Операционная зрелость.** PostgreSQL с ACID-гарантиями является наиболее консервативным и знакомым команде выбором для самого чувствительного к ошибкам домена системы.

## Итоговая карта polyglot persistence

| Тип технологии | Сервисы | Причина группировки |
|---|---|---|
| PostgreSQL (реляционная) | Сервис ставок, Сервис кампаний, Сервис финансов | Стабильная структура данных, потребность в консистентности от causal до строгой ACID |
| ClickHouse (колоночная) | Сервис статистики, Сервис аналитики | Write-heavy или аналитически тяжёлый профиль нагрузки, eventual consistency допустима |
| Redis (key-value) | Кэш перед сервисом ставок | Максимально быстрый доступ к горячим данным таргетинга без сложных запросов |
| Kafka (event streaming) | Инфраструктура обмена между всеми сервисами | Гарантия доставки, повторное чтение, история событий для аудита финансовых операций |

Использование разных технологий для разных сервисов является сознательным решением, а не разрастанием технологического стека: каждый выбор обоснован собственным профилем нагрузки и требованиями к консистентности конкретного сервиса, а не единым шаблоном на всю систему.
